# NOTES

Best config: a 4-layer / 4-head / 160-dim GPT (1.59M params) trained for 2000 steps on a byte-level BPE tokenizer (vocab 1024) with AdamW, a warmup→cosine LR schedule (peak 1.5e-3, warmup 100), batch 8, block 128. Final dev bpb 1.9996, down from the 2.3718 baseline.

The two changes that mattered were the LR schedule and the tokenizer. The baseline was badly undertrained — its loss was still falling at step 2000 — so replacing the flat 3e-4 LR with a warmup→cosine schedule at a higher peak alone cut bpb by 0.18. The BPE tokenizer cut another 0.19 because at ~2.3 bytes/token it packs ~2.3× more bytes of context into the same 128-token window and lets the model see ~2.3× more effective bytes per step, which is exactly what a step-capped run is starved for. This matters most for the Hindi text, where the byte tokenizer spent 3 tokens per Devanagari character. The BPE stays lossless (all 256 byte values are base tokens with a byte fallback), so the per-byte metric remains valid. The "textbook" GPT-2 tricks — weight tying, init_std 0.02, residual scaling, weight decay 0.1 — all *hurt* here, raising bpb to ~2.17-2.20, because in a tiny model with only 2000 steps they shrink the early signal and slow optimization rather than regularize an overfit. Pushing the LR higher (2.5e-3) gave no further gain, confirming the model was near its optimization limit for this budget. Net: the win came from feeding the model more information per step (tokenizer) and actually letting it converge (schedule), not from architectural cleverness.
